# Content-Addressed Versioning

Every definition in March is identified by a BLAKE3 hash of its typed intermediate representation. Names are human-readable aliases pointing to hashes. There are no version numbers — the hash is the version.

This document specifies the hash-first name resolution approach: hashes are computed during TIR lowering and become the primary identity for definitions throughout the rest of the pipeline (monomorphization, defunctionalization, LLVM emission).

> **Note:** This spec supersedes the content-addressing section in `design.md` (lines 575-597), which describes AST-level SHA-256 hashing. This spec uses TIR-level BLAKE3 hashing with sig/impl hash separation. `design.md` should be updated to match once this spec is approved. The pinned import prefix changes from `sha256:` to `blake3:`.

## Goals

- **Incremental compilation**: skip recompiling definitions whose hashes haven't changed
- **Reproducible builds**: same source → same hashes → same output, always
- **Package/dependency management foundation**: publish libraries as hash-addressed definitions, eliminate version conflicts and the diamond dependency problem

## Core Hashing Primitives

Every TIR definition gets two BLAKE3 hashes:

- **Signature hash (`sig_hash`)**: the public interface only — name, parameter types, return type, visibility. For types: name, type parameters, variant/record structure. A sig_hash change means dependents must recompile.
- **Implementation hash (`impl_hash`)**: the full TIR body including the signature. An impl_hash change with unchanged sig_hash means dependents are unaffected.

Hashes are computed over a canonical serialization of the TIR node — deterministic, stable, no spans, no source locations, no comments. Structural content only.

```
sig_hash(fn map) = BLAKE3(name="map", params=[TVar "a", TFn([TVar "a"], TVar "b"), TList(TVar "a")], ret=TList(TVar "b"))
impl_hash(fn map) = BLAKE3(sig_hash ++ body_bytes)
```

### Types

```ocaml
type content_hash = Blake3 of string  (* 32-byte digest, hex-encoded for display *)

type def_id = {
  did_name : string;          (* human-readable, for errors/display *)
  did_hash : content_hash;    (* the actual identity — uses impl_hash *)
}

type def_kind =
  | FnDef of fn_def
  | TypeDef of type_def
  | ActorDef of actor_def
  | ImplDef of impl_def
  | SigDef of sig_def

type hashed_def = {
  hd_sig_hash  : content_hash;
  hd_impl_hash : content_hash;
  hd_def       : def_kind;
}
```

### Hashing by Definition Kind

**Functions (`DFn`):** sig_hash covers name, parameter types, return type, visibility. impl_hash covers the full TIR body.

**Types (`DType`):** sig_hash covers name, type parameters, and the full structure (variant constructors with their field types, record field names and types, alias target). For types, sig_hash and impl_hash are identical — types have no "private body" distinct from their interface. A type change always propagates to dependents.

**Actors (`DActor`):** An actor definition is decomposed into multiple hashed definitions:
- The **actor type** (state fields, message variant type) is hashed as a type definition.
- Each **message handler** (`on` clause) is hashed as a function definition.
- The **init function** is hashed as a function definition.
- The **dispatch function** (generated by lowering) is hashed as a function definition.
- The **spawn function** (generated by lowering) is hashed as a function definition.
- By convention, all parts of an actor are grouped as a single SCC for hashing, even if the dependency graph wouldn't strictly require it. This simplifies the model: an actor is one atomic unit from the CAS perspective. Changing any part of an actor invalidates the whole actor.

**Interface implementations (`DImpl`):** An `impl Eq(Int)` block is hashed as a standalone definition. Its sig_hash is derived from the interface name, the implementing type, and the method signatures. Its impl_hash covers the full method bodies.

- When an `impl` block's impl_hash changes but sig_hash doesn't: no downstream invalidation (internal change only).
- When an `impl` block's sig_hash changes: callers using the interface constraint on that type are invalidated.
- Coherence: at most one `impl` per (interface, type) pair. Two versions of the same `impl` in the CAS from different builds is fine — they're different hashes, and only one is "current" per the lockfile. Coherence is enforced at build time when resolving the active set of impls, not at the CAS storage level.

**Module signatures (`DSig`):** A `sig` block gets its own hash, separate from the module's definitions. When a module has an explicit `sig`, downstream modules depend on the `sig` block's hash rather than individual function sig_hashes. This is the "downstream code depends on the sig, not the implementation" guarantee from the design spec. If no explicit `sig` exists, downstream modules depend on individual function sig_hashes (the inferred interface).

## Hash-First Name Resolution

Definitions are identified by `def_id` (name + content hash) throughout the TIR and beyond. This replaces raw string-based name resolution.

### TIR Module Structure

```ocaml
module DefMap = Map.Make(String)

(* replaces the current flat tir_module *)
type tir_module = {
  tm_name  : string;
  tm_sccs  : scc list;              (* topologically sorted *)
  tm_index : def_id DefMap.t;       (* name → def_id, deterministic iteration *)
}
```

Note: `tm_index` uses an ordered `Map`, not `Hashtbl`, to ensure deterministic iteration if ever traversed during serialization or diagnostics.

### Atom-Level References

Top-level name references carry a `def_id`. Local variables remain plain strings.

```ocaml
type atom =
  | AVar of var                  (* local variable, unchanged *)
  | ADefRef of def_id            (* reference to a top-level definition *)
  | ALit of literal
```

### Resolution During Lowering

1. Lower a definition to TIR → compute its hashes → register `name → def_id` in `tm_index`
2. When lowering a reference to a name → look up in `tm_index` → emit `ADefRef(def_id)`
3. Forward references (mutual recursion): handled by SCC-based group hashing (see below)

## SCC-Based Group Hashing

Mutually recursive definitions can't be hashed individually — `even` references `odd` and vice versa. The solution: detect strongly-connected components in the definition dependency graph, then hash each SCC as a unit.

### Dependency Analysis

1. Build a directed graph: each definition is a node, each reference to another definition is an edge (both value-level references and type-level references)
2. Run Tarjan's algorithm to find SCCs
3. Topologically sort the SCCs — guaranteed acyclic at the SCC level

### SCC Types

```ocaml
type scc =
  | Single of hashed_def
  | Group of {
      grp_hash : content_hash;          (* BLAKE3 of all members combined *)
      grp_defs : hashed_def list;       (* individual defs with own hashes *)
    }
```

### Hashing Rules

- **Singleton SCC (no self-recursion):** hash normally. Sig hash from signature, impl hash from full body. References to already-hashed definitions use their `def_id`.

- **Singleton SCC (self-recursive):** the canonical serialization replaces self-references with a `SelfRef` marker. Self-references are identified by name match against the definition being hashed (before the hash exists). If a local variable shadows the function name, the local binding takes precedence per normal scoping — only unresolved top-level references to the function's own name become `SelfRef`. The hash is stable because the serialized form is deterministic.

- **Multi-definition SCC:** all members hashed together as a group:
  1. Sort members by name (deterministic ordering)
  2. Serialize each member's canonical form with intra-group references as `GroupRef(member_name)` — a placeholder meaning "another member of my SCC"
  3. Concatenate all serialized members
  4. BLAKE3 the concatenation → `grp_hash`
  5. Each individual member gets `impl_hash = BLAKE3(grp_hash ++ member_name)` for per-definition identity within the group

### Sig Hashes for SCC Group Members

Sig hashes for group members may also require group hashing when mutually recursive types are involved. For example:

```march
type Tree = Leaf | Node(Forest)
type Forest = Empty | Trees(Tree, Forest)
```

The signature of `Tree` references `Forest` and vice versa, creating a cycle at the signature level.

**Rules:**
- If an SCC's signatures do not reference each other (e.g., mutually recursive functions with independent types like `even: Int -> Bool` and `odd: Int -> Bool`), sig hashes are computed independently per member.
- If an SCC's signatures do reference each other (e.g., mutually recursive types, or functions whose parameter/return types are mutually recursive types within the same SCC), sig hashes use the same group-hashing mechanism: `GroupSigRef(member_name)` placeholders, concatenation, BLAKE3. Each member's sig_hash is then `BLAKE3(group_sig_hash ++ member_name)`.

Detection: after computing the SCC, build a secondary dependency graph over signatures only (type references in parameter/return types). If this sub-graph has cycles within the SCC, use group sig hashing. Otherwise, compute sig hashes independently.

## Content-Addressed Store

Maps hashes to serialized TIR definitions and compiled artifacts. Two layers: project-local is authoritative, global is a read-through cache.

### On-Disk Layout

```
.march/cas/                          # project-local (authoritative)
  objects/
    a3/f4b2c1...                     # 2-char hex prefix directories (git-style)
  artifacts/
    a3/f4b2c1...$aarch64-darwin.o    # compiled artifacts keyed by compilation_hash
  index.bin                          # name → def_id mapping for current build

~/.march/cas/                        # global (shared across projects)
  objects/
    a3/f4b2c1...
  artifacts/
    a3/f4b2c1...$aarch64-darwin.o
```

### Store Interface

```ocaml
type compilation_hash = Blake3 of string
(** impl_hash + target_triple + compiler_flags *)

module Cas : sig
  type t

  val create : project_root:string -> t
  (** Opens project-local store, discovers global store at ~/.march/cas/ *)

  val lookup_def : t -> content_hash -> hashed_def option
  (** Look up a TIR definition. Check project-local first, then global. *)

  val store_def : t -> hashed_def -> unit
  (** Write a TIR definition to project-local store. *)

  val lookup_artifact : t -> compilation_hash -> string option
  (** Look up compiled artifact (LLVM IR or object file path). *)

  val store_artifact : t -> compilation_hash -> string -> unit
  (** Store compiled artifact. *)

  val compilation_hash : content_hash -> target:string -> flags:string list -> compilation_hash
  (** Derive compilation hash from impl_hash + target triple + flags. *)

  val lookup_name : t -> string -> def_id option
  (** Resolve a human-readable name to its current def_id via index *)

  val update_index : t -> (string * def_id) list -> unit
  (** Bulk update the name→def_id index after a build *)

  val gc : t -> keep_defs:content_hash list -> keep_artifacts:compilation_hash list -> int
  (** Remove objects not in keep_defs and artifacts not in keep_artifacts.
      Returns total count removed. Callers derive keep_artifacts from
      keep_defs × active target/flag combinations. *)
end
```

### Lookup Strategy

1. Check project-local `objects/`
2. On miss, check global `~/.march/cas/objects/`
3. On hit in global, optionally copy to project-local (warm local cache)
4. On miss in both, definition needs compilation

### Canonical Serialization

A purpose-built binary encoding in `lib/cas/serialize.ml`. Not `Marshal` (not deterministic across OCaml versions), not protobuf (unnecessary complexity).

Requirements:
- **Deterministic**: same TIR always produces same bytes
- **Stable**: format changes are versioned so old objects remain readable
- **Fast**: runs on every definition every build

Object format:

```
[version: u8] [kind: u8] [sig_hash: 32B] [impl_hash: 32B] [payload: ...]
```

`kind` distinguishes the storage-level form: single function, single type, impl, sig, SCC group. Actors are decomposed before storage — their parts are stored as individual function/type definitions within an SCC group object.

#### Payload Encoding Rules

All multi-byte integers are little-endian. The encoding is prefix-free (every construct is unambiguously decodable without lookahead).

| TIR construct | Encoding |
|---|---|
| Integers | Fixed-width i64 (8 bytes) |
| Floats | IEEE 754 double (8 bytes) |
| Strings | u32 length prefix + UTF-8 bytes (no null terminator) |
| Booleans | Single byte: 0x00 = false, 0x01 = true |
| Type constructors | u8 tag (unique per TIR type variant) + fields in declaration order |
| Record fields | Sorted alphabetically by field name before serialization |
| Lists/sequences | u32 count + elements in order |
| `ADefRef(def_id)` | Tag 0x80 + 32-byte hash (the `did_hash`) |
| `SelfRef` | Tag 0x81 |
| `GroupRef(name)` | Tag 0x82 + string-encoded name |
| `GroupSigRef(name)` | Tag 0x83 + string-encoded name |
| `TVar(name)` | Tag 0x90 + string-encoded name (preserved for polymorphic defs) |

Type constructor tags are assigned sequentially starting from 0x01 and documented in a tag registry within `serialize.ml`. Adding a new TIR node type means assigning the next available tag and bumping the format version.

#### Format Versioning and Migration

Multiple format versions can coexist in the same store. On read, the version byte selects the appropriate deserializer. When the format version changes:
- Old objects remain readable (deserializers for all prior versions are kept)
- New objects are written in the current format version
- No automatic migration — old objects are lazily re-hashed and re-stored when their definitions are recompiled
- A `march cas migrate` command can bulk-rewrite objects to the current format (optional, for store compaction)

## Pipeline Integration

### Revised Pipeline

```
Parse → Desugar → Typecheck → Lower to TIR
  → Dependency analysis (build definition graph)
  → SCC detection (Tarjan's)
  → Topological sort
  → Hash each SCC (canonical serialize → BLAKE3)
  → CAS lookup (hit? skip recompilation of this SCC)
  → Monomorphize (keyed by def_id + type_args)
  → Defunctionalize
  → LLVM emit
```

### Cache-Hit Fast Path

When a `def_id` already exists in the CAS with cached compilation artifacts, skip everything downstream for that SCC:

```ocaml
(* scc_impl_hash returns impl_hash for Single, grp_hash for Group *)
let compile_scc store ~target ~flags scc =
  let ch = Cas.compilation_hash (scc_impl_hash scc) ~target ~flags in
  match Cas.lookup_artifact store ch with
  | Some artifact_path -> artifact_path
  | None ->
    let mono = Mono.run scc in
    let defun = Defun.run mono in
    let llvm = Llvm_emit.run defun in
    let path = write_object_file llvm in
    Cas.store_def store (scc_hashed_def scc);
    Cas.store_artifact store ch path;
    path
```

### Monomorphization

Processes one SCC at a time (currently processes all functions as a single worklist). Cache key changes from `(fn_name, type_args)` to `(def_id, type_args)`. Mangled names embed a hash prefix: `map$a3f4$Int` rather than `map$Int`, guaranteeing link-time uniqueness when multiple versions coexist.

### Defunctionalization

Minimal changes. Lambda lifting produces new definitions that get hashes derived from `BLAKE3(parent_impl_hash ++ lambda_index)`, keeping them deterministic.

### LLVM Emission

Function names use a sanitized hash-based name. `@march_main` remains a special case looked up by name. All other cross-references use hash-prefixed mangled names for link-time uniqueness.

### REPL / JIT

The JIT in `repl_jit.ml` currently tracks compiled function names in a `Hashtbl`. With content addressing, the key becomes `impl_hash`: if a definition's hash hasn't changed since the last REPL fragment, skip recompilation. Strictly better than name-based checks — detects when a function's body changed but its name didn't.

## Lockfile

`march.lock` is committed to the repo. Records the resolved `def_id` for every name after a successful build. Given the same lockfile, builds are reproducible.

### Format

```
# march.lock — auto-generated, do not edit
# format: name sig_hash impl_hash [group_hash]
List.map       b7e2a1f3...  d4c8e9a0...
List.filter    91f0c3b2...  a2d7f1e4...
Option.map     b7e2a1f3...  8c3a5b20...
main           e1a4d902...  f7b3c6a1...
Tree           c4a1f2e7...  c4a1f2e7...  grp:7f8e9d01...
Forest         d3b2a1c8...  d3b2a1c8...  grp:7f8e9d01...
```

SCC group members include their `grp_hash` as an optional fourth column. This allows reconstructing SCC membership from the lockfile for reproducibility verification — if two definitions share a `grp:` hash, they were in the same SCC.

### Pinned Imports

```march
use Collections.insert                     -- resolves via lockfile
use blake3:a3f4b2c1... as insert           -- pinned to exact hash
```

The lockfile handles the first form. The second form is an explicit override — the compiler verifies the hash exists in the CAS and uses it directly, skipping name resolution.

Pinned hash literals are always the full 64-character hex string in source code (no short prefixes — ambiguity is unacceptable in source). The parser distinguishes `blake3:...` from qualified module paths by the `blake3:` prefix keyword, which is reserved and cannot be used as a module name.

### Update Rules

- `march build`: recompute hashes for changed definitions, update lockfile, write new objects to CAS
- Unchanged source → unchanged hash → unchanged lockfile entry
- Sig hash change → walk dependents transitively, recompute their hashes
- Lockfile diffs in version control show exactly what changed

## Error Reporting and Developer Experience

Hashes are for machines. Developers interact with names.

### Error Messages

All diagnostics use `did_name`. Hashes only appear for disambiguation.

```
Error: Type mismatch in List.map
  expected: (a -> b) -> List(a) -> List(b)
  got:      (Int -> String) -> List(Float) -> List(String)
```

When two versions of the same name coexist:

```
Error: Ambiguous reference to 'insert'
  insert (blake3:a3f4b2..) from Collections
  insert (blake3:91f0c3..) from LegacyUtils
```

Short hash prefixes auto-expand to minimum unambiguous length, like git commit hashes.

### Build Output

```
$ march build
  [cached]  List.map (sig unchanged)
  [cached]  List.filter (sig unchanged)
  [rebuild] Parser.parse (impl changed)
  [rebuild] Parser.tokenize (depends on Parser.parse sig)
  [done]    4 cached, 2 rebuilt
```

### REPL

```
march> fn double(x) = x * 2
  [new] double: blake3:c4a1f2...

march> fn double(x) = x + x
  [updated] double: blake3:e7b3d1... (was blake3:c4a1f2...)
```

### Lockfile Diffs in Code Review

```diff
- Parser.parse  a1b2c3...  d4e5f6...
+ Parser.parse  a1b2c3...  7f8e9d...   # impl changed, sig stable
```

## Migration Strategy

The transition from the current flat `tir_module` to SCC-based hash-addressed structure is incremental:

1. **Phase 1: Add canonical serialization.** Implement `lib/cas/serialize.ml` with deterministic TIR encoding. No pipeline changes — serialization is a pure function from TIR to bytes, testable in isolation.

2. **Phase 2: Add hashing.** Compute BLAKE3 hashes after lowering but before monomorphization. Attach hashes to definitions as metadata. The existing pipeline continues to work — hashes are computed but not yet used for identity or caching.

3. **Phase 3: Add CAS store.** Implement `lib/cas/cas.ml` with project-local and global stores. Store hashed definitions after each build. Still no pipeline behavior change — the store is populated but not consulted.

4. **Phase 4: Add dependency analysis and SCC detection.** Implement Tarjan's, topological sort, reorder `tir_module` to SCC list. Monomorphization switches from flat worklist to per-SCC processing.

5. **Phase 5: Hash-first name resolution.** Introduce `ADefRef`, `def_id`, replace string-based references in TIR. This is the "big bang" change but by this point all supporting infrastructure is in place and tested.

6. **Phase 6: Cache-hit fast path.** Consult the CAS before monomorphization. Skip downstream passes for cache hits.

Each phase is independently testable and deployable. The compiler works correctly (just without caching) at every intermediate phase.

## Future Work (Not V1)

- **Name registry**: shared registry mapping `package/name` → `def_id` for package publishing. The "no version numbers" promise.
- **Distributed cache**: remote CAS (S3, HTTP) for CI cache sharing across machines.
- **Semantic code search**: find definitions by type signature across the content-addressed store.
- **Actor migration**: actors serialized with code identified by hash, fetched from CAS on remote nodes.
